🎯 KEY COMPONENTS BREAKDOWN
============================

1. API ENDPOINTS (main.py)
   ├── POST /api/pipeline/ingest     → Receive clauses from pipeline
   ├── GET  /api/analyze/{job_id}    → Trigger 4-agent analysis
   ├── WS   /ws/{job_id}             → Stream real-time events
   ├── GET  /api/report/{job_id}     → Download full JSON report
   └── GET  /health                  → Health check for Cloud Run

2. AGENT PIPELINE (orchestrator.py)
   ├── Step 1: Extractor             → Classify clause type (8 categories)
   ├── Step 2: Analyst               → Score 5 risk dimensions (0-10)
   ├── Step 3: Adversary ★USP        → Hostile lawyer weaponization analysis
   └── Step 4: Advisor               → Plain language + recommendations

3. DATA MODELS (models.py)
   ├── Clause                        → Input clause structure
   ├── IngestRequest                 → API request schema
   ├── RiskScores                    → 5-axis risk scores (validated 0-10)
   ├── ClauseAnalysis                → Complete analysis result
   ├── AnalysisReport                → Full job report
   └── WebSocketEvent                → Real-time event structure

4. STORAGE (storage.py)
   ├── create_job()                  → Initialize analysis job
   ├── update_job_status()           → Track processing state
   ├── add_clause_analysis()         → Store completed analysis
   ├── get_job()                     → Retrieve full report
   └── job_exists()                  → Check job existence

5. SECURITY (config.py)
   ├── Secret Manager integration    → Runtime secret retrieval
   ├── LRU caching                   → Performance optimization
   ├── No hardcoded secrets          → All from Secret Manager
   └── Error handling                → Never log secrets


🔐 SECURITY ARCHITECTURE
=========================

Layer 1: Network
├── Cloud Load Balancer with IAP
├── Internal ingress only
└── TLS 1.2+ encryption

Layer 2: Authentication
├── --no-allow-unauthenticated

Layer 3: Application

Layer 5: Container
├── Non-root user (UID 1000)

└── Vulnerability scanning

📊 DATA FLOW
=============

Ingestion:
Client → POST /api/pipeline/ingest → Firestore (pending)

Analysis:
Client → GET /api/analyze/{job_id} → Background Task
  ├── Update status: processing
  ├── For each clause:
  │   ├── Extractor → emit event
  │   ├── Analyst → emit event
  │   ├── Adversary → emit event
  │   ├── Advisor → emit event
  │   └── Store result → emit complete
  └── Update status: completed

Real-time:
Client → WS /ws/{job_id} → Stream events as they occur

Report:
Client → GET /api/report/{job_id} → Firestore → JSON response

🌟 UNIQUE SELLING POINT
========================

The Adversary Agent (agents/adversary.py)

System Prompt:
"You are a hostile lawyer hired by the counterparty. Argue how this 
clause could be weaponized against the user in the worst realistic 
scenario. Be specific, cite realistic legal outcomes."

What It Provides:
├── Hostile interpretation of clauses
├── Specific weaponization scenarios
├── Realistic legal outcomes
├── Precedent citations
├── Procedural advantages for counterparty
└── Hidden obligations and restrictions

Why It Matters:
Most tools just highlight risks. LexGuard shows EXACTLY how a hostile 
lawyer would exploit each clause, providing unprecedented insight into 
contract vulnerabilities.
